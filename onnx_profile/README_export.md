# Gemma HF To ONNX Export Flow

This note covers the path from a raw Hugging Face Gemma directory to an ONNX model that can be used by the repository's ONNX Runtime profiling scripts.

## Scope

- Input: raw Hugging Face directory with `config.json`, tokenizer files, and `*.safetensors`
- Output: exported ONNX directory plus graph inspection artifacts
- Non-goal: building the local `onnxruntime` source tree before profiling

## Why This Step Exists

`onnxruntime` cannot execute a raw Hugging Face `safetensors` directory directly. The model must first be exported to ONNX, then the exported graph must be checked to see whether it includes `past_key_values` or other cache inputs/outputs.

## Recommended Flow

1. Inspect the raw model directory.

```bash
nix develop -c just hf-inspect model_dir=/home/monad/develop/ai_accel/gemma3-1B
```

2. Prepare an isolated Python environment for export tooling.

```bash
python3 -m venv .venv-onnx-export
. .venv-onnx-export/bin/activate
pip install -r onnx_profile/requirements.txt
```

3. Check the export plan without running the heavy export.

```bash
nix develop -c just gemma-onnx-export-dry model_dir=/home/monad/develop/ai_accel/gemma3-1B
```

4. Run the export only when you are ready.

```bash
nix develop -c just gemma-onnx-export model_dir=/home/monad/develop/ai_accel/gemma3-1B
```

5. Inspect the exported ONNX graph.

```bash
nix develop -c just onnx-inspect model=/home/monad/develop/ai_accel/gemma3-1B-onnx/model.onnx
```

## What To Look For In The ONNX Graph

- Presence of `past_key_values`, `past_key`, `past_value`, `present`, or `cache` inputs/outputs
- Whether the export uses external data files in addition to the main `.onnx`
- Whether `config.json` and tokenizer files were copied into the export directory

Interpretation:

- If past-KV graph inputs and outputs exist, the profiling flow can attempt decode cache reuse.
- If they do not exist, only prefill or whole-graph measurements should be treated as supported by the export.

## Package Notes

The default Nix shell in this repository intentionally stays lighter than a full export environment. Export-related packages are listed in `onnx_profile/requirements.txt` so you can install them into a dedicated virtual environment without changing the FPGA workflow.

Expected export packages:

- `transformers`
- `optimum`
- `torch`
- `safetensors`
- `onnx`
- `onnxruntime`
- `psutil`

## Failure Policy

- If the raw model directory is missing required files, the inspection step fails clearly.
- If export dependencies are missing, the dry-run step reports exactly which packages are absent.
- If ONNX export fails, the wrapper script prints the attempted command and the captured error output.
- No benchmark numbers or fake profiling outputs are generated by the export helper.
